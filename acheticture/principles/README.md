# Principles

This directory contains a small set of stable architectural principles used to guide decisions across the system.

Principles should be:

- durable
- short
- easy to apply
- few in number

Good principle statements are concise and actionable, for example:

- prefer context boundaries over repository boundaries
- prefer explicit contracts over implicit coupling
- keep canonical terminology consistent across docs and code

Do not turn this directory into a catalog of preferences. If a rule is narrow or situational, it likely belongs in `cross-cutting/` or an ADR instead.
