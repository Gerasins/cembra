## Вступление — цель презентации (0:30)

- Почему мигрируем: уменьшить ручной боилерплейт, упростить интеграцию схем, улучшить консистентность.
- Что ожидаем: меньше ручных резолверов, автоматическое связывание типов, требования к согласованности схем.

---

## Stitching — кратко (3:00)

- Концепция: «возьми этот резолвер и приклей его к типу».
- Управление резолверами вручную, явная делегация запросов.

Notes:
- Показать небольшой пример: remote schema + добавленный resolver для поля `user.profile`.

---

## Fusion — основные идеи (4:00)

- Fusion склеивает одинаковые типы между схемами по имени.
- Подход: не привязывать резолверы вручную, а объявлять источники данных (lookup'ы).

---

## Терминология Fusion (3:30)

- `lookup` — query, которая возвращает тип (в Fusion это lookup).
- `inner` lookup — невидимы внешнему потребителю, используются только внутри Fusion.
- `external` lookup — видимы всем, authoritative источник данных.

Notes:
- Объяснить разницу на диаграмме: источники → Fusion → объединённая схема.

---

## Как Fusion выбирает lookup (4:00)

- Fusion смотрит, какие данные запрашиваются клиентом, и выбирает подходящий lookup.
- Самостоятельно решает, какой lookup использовать (external/inner) для заполнения полей.

---

## Как Fusion склеивает одинаковые типы (4:00)

1. Находит одинаковые названия типов в разных схемах.
2. Сравнивает свойства полей (имя и тип).
3. Если свойства совпадают — берёт `external` lookup и заполняет поля для `inner` lookup.
4. Мы можем обратиться к этим lookup'ам и использовать результаты в inner резолверах.

Notes:
- Подчеркнуть: согласованность имен и типов критична.

---

## Пример (демо) — схема и запрос (2:30)

```graphql
# Источник A
type User { id: ID!, name: String }
query { userById(id: ID!): User }

# Источник B
type User { id: ID!, email: String }
query { userEmailById(id: ID!): User }
```

Коротко: Fusion увидит `User` в обоих источниках, сравнит поля и составит unified `User`.

Пример lookup'ов: `userById` (external), `userEmailById` (inner/external в зависимости от конфига).

## Риски и mitigations (2:30)

- Конфликты свойств: привести схемы к совместимости или задать явные трансформации.
- Неожиданный выбор lookup'а: логирование, трассировка, явное указание authoritative.
- Авторизация: убедиться, что правила применяются на уровне lookup/resolver.

## Тесты и мониторинг (1:30)

- Контрактные тесты схем (CI) на совпадение полей/типов.
- Трассировка lookup'ов, latency, error rates.
- DataLoader/кэширование для борьбы с N+1.

## Grill-me / Q&A (2:00)

- Как Fusion решает конфликт типов? — Требуется ручная конфигурация или authoritative lookup.
- Что если external недоступен? — Настроить fallback или graceful degradation.
- Как дебажить источник поля? — Включить трассировку lookup'ов и логирование выбора.
- Можно ли комбинировать Stitching и Fusion? — Да, поэтапная миграция возможна.
